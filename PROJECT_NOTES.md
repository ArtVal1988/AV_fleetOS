## Що це за проєкт

Система управління автопарком для прокату авто (av-fcs.com, пов'язана з
TOPRENT). Український інтерфейс. Vanilla JS фронтенд в одному HTML-файлі
(без збірки/бандлера), Express + SQLite бекенд.

## Структура репозиторію

```
public/index.html   — увесь фронтенд (HTML+CSS+JS в одному файлі, ~8000+ рядків)
server/server.js    — Express застосунок, точка входу
server/db.js        — схема SQLite (better-sqlite3 / node:sqlite)
server/routes/       — auth.js, bookings.js, vehicles.js, stickers.js, settings.js, price-check.js
```

## Сервер (VPS) — структура папок

- `/var/www/AV_fleetOS` — git-клон репозиторію (сюди робиться `git pull`)
- `/var/www/AV_fleetOS-server` — **прод**, робоча копія, яку реально обслуговує pm2-процес `AV_fleetOS` (порт 3000, av-fcs.com)
- `/var/www/AV_fleetOS-staging` — **staging**, копія прод-версії (окрема БД!) на порту 3001, pm2-процес `AV_fleetOS-staging`
  - Доступ: `http://173.242.58.173:3001` (пряме IP:порт, без домену)
  - `.env` там ідентичний прод, окрім `PORT=3001`

## Робочий процес деплою (ВАЖЛИВО!)

1. Claude вносить правки локально в контейнері, комітить і пушить у GitHub (`git push origin main`)
2. Спершу деплоїмо **тільки на staging**:
   ```
   bash /root/update-staging.sh
   ```
3. Користувач тестує на `http://173.242.58.173:3001`
4. Тільки після явного підтвердження користувача — деплой на прод:
   ```
   bash /root/update.sh
   ```
5. **Завжди перевіряти після деплою** контрольними сумами:
   ```
   md5sum /var/www/AV_fleetOS-server/public/index.html /var/www/AV_fleetOS-server/server.js
   ```
   (для staging — аналогічно, шлях `/var/www/AV_fleetOS-staging/...`)

**У Claude немає SSH-доступу до сервера.** Усі команди деплою виконує
користувач вручну, Claude лише формулює команди й перевіряє результат.

**GitHub токен** протухає за кілька днів — якщо `git push` видає помилку
автентифікації, попросити користувача згенерувати новий Personal Access
Token (classic, scope `repo`) і оновити remote:

## Важливі уроки цієї сесії (щоб не наступати на ті самі граблі)

1. **Ніколи не залишай непарні `<div>`.** Одного разу забутий закриваючий
   тег зламав усю верстку форми продовження оренди. Після будь-якої правки
   HTML-структури — перевіряти баланс тегів:
   ```python
   opens = html.count('<div'); closes = html.count('</div>')
   ```

2. **Кешування.** Express за замовчуванням додає ETag і не забороняє кеш
   статичних файлів — це спричиняло 304-відповіді й "залипання" старої
   версії сайту після деплою. Вже виправлено: `app.set('etag', false)` +
   `Cache-Control: no-store` на всі `/api/*` та на `public/index.html`.
   Якщо після деплою сайт показує старе — перше, що перевіряти.

3. **applyOrderLockState() залишає "мертві зони".** Ця функція виставляє
   inline `pointer-events:auto` на `#bm-block-1` (і подібні), щоб блок
   клієнта лишався клікабельним навіть при заблокованому замовленні. Це НЕ
   скидалось при закритті модалки — невидиме закрите вікно "ловило" кліки
   в тому місці екрана. Виправлено в `closeBookingModal()` — там тепер явно
   скидаються ці inline-стилі.

4. **mini-picker (z-index:400) вищий за modal-overlay (z-index:300).**
   Якщо `mini-picker` (спливаюча панель на порожній клітинці календаря)
   лишається відкритою і потім відкривається картка замовлення — mini-picker
   "просвічує" крізь неї. Виправлено викликом `closeMiniPicker()` на
   початку `editBooking()` і `openBookingModal()`.

5. **Нативний `<input type="date">`/`type="time"` в Chrome має дивний баг**
   всередині прокручуваних модалок: календар/час може "зникати-з'являтися"
   при переході фокуса між полями (відтворюється навіть в Інкогніто, навіть
   в Edge, навіть на секціях, які не редагувались). `blur()`/`preventDefault`
   на mousedown не завжди допомагають. **Рішення: власний JS-календар**
   (`openCustomDatePicker`/`openCustomTimePicker`, клас `.custom-dp-trigger`)
   замість нативного — вже реалізовано для полів Період/Отримання/
   Продовження/Повернення (дата+час). Значення полів лишаються в тому ж
   форматі (`YYYY-MM-DD`, `HH:MM`), усі існуючі `onchange`/`oninput`
   обробники продовжують працювати без змін.

6. **Після кожної правки HTML/JS — синтаксис-чек перед пушем:**
   ```python
   scripts = re.findall(r'<script>(.*?)</script>', html, re.S)
   # записати в тимчасовий .js і прогнати node --check
   ```
   Це кілька разів рятувало від деплою зламаного коду.

7. **Euro (€) прибрано скрізь, окрім поля "Депозит"** (свідоме рішення
   користувача) — валюта в "Вартість/Курс" продовження, "Загалом",
   "Оплачено" і основних "Фінансах" тепер лише $ і ₴.

## Незавершене / потребує уваги

- **`/api/price-check`** (routes/price-check.js) — новий маршрут, парсить
  сторінку авто на toprent.ua (regex по тексту, без cheerio) і повертає
  тарифи+заставу+характеристики. Побудований на основі структури сторінки
  `https://toprent.ua/en/skoda-rapid-2015`, яку Claude бачив через
  web_fetch (НЕ через сервер напряму — важливо, бо може відрізнятись від
  сирого HTML). **Потребує реального тестування** на staging з живим
  посиланням користувача — regex могли не влучити в точну розмітку.
  Allowlist хостів: лише `toprent.ua`/`www.toprent.ua` (захист від SSRF).

- **Досі під питанням:** чи повністю зникла проблема з нативними полями
  дат/часу після переходу на власний календар (п.5 вище) — просили
  користувача перевірити після останнього деплою, відповідь ще не отримана.
